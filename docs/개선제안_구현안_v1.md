## 0. 통합 마이그레이션 `0003_improvements.sql` (개요)

각 항목 상세에서 쓰는 스키마 변경을 한 파일로 모은다. (적용: Supabase SQL Editor에 붙여넣고 Run, 0002 이후 1회)

```sql
-- 캘박 0003: 개선 4건 스키마

-- ── A1: 팬아웃 비동기 + 안전망 ───────────────────────────
alter table photos
  add column staging_path text,                 -- Supabase Storage 임시 원본 경로
  add column fanout_done  boolean not null default false;

alter table photo_drive_copies
  add column error_code text,                    -- 'quota' | 'auth' | 'other'
  add column attempts    integer not null default 0,
  add column updated_at  timestamptz not null default now();

-- ── A2: 날씨 정확도 표식 ─────────────────────────────────
alter table weather_snapshots
  add column is_approx boolean not null default false;  -- 좌표 추정/폴백 여부

-- ── B5: 알림 ─────────────────────────────────────────────
create table notifications (
  id         uuid primary key default gen_random_uuid(),
  user_id    uuid not null references auth.users(id) on delete cascade, -- 받는 사람
  memory_id  uuid references memories(id) on delete cascade,
  actor_id   uuid references auth.users(id) on delete set null,        -- 행위자
  type       text not null,                       -- 'comment'|'photo'|'on_this_day'
  preview    text,                                 -- "댓글: 그 횟집 진짜였다"
  read_at    timestamptz,
  created_at timestamptz not null default now()
);
create index on notifications(user_id, created_at desc);

alter table notifications enable row level security;
create policy notifications_self on notifications
  for select using (user_id = auth.uid());
create policy notifications_self_update on notifications      -- 읽음 처리
  for update using (user_id = auth.uid()) with check (user_id = auth.uid());
-- INSERT(알림 생성)는 서버 service role 전용 → 정책 없음(RLS가 차단, admin이 우회)
```

> B4(1년 전 오늘)는 **스키마 변경 없음**(기존 `memories.date`만 사용).

## A1. 사진 팬아웃 비동기화 + 영속 안전망 ⭐

### 문제 (현재 코드)
- `web/app/api/memories/[id]/photos/route.ts`가 `replicatePhoto`를 **await** → 멤버 N명이면 Drive 업로드 N회를 업로더가 전부 기다린다(모바일 체감 지연·함수 타임아웃 위험).
- `replicatePhoto`(photos.ts:61)는 원본 바이트를 **어디에도 보관하지 않는다.** 팬아웃이 전부 `skipped/failed`면 복사본 0개짜리 죽은 `photos` 레코드가 남는다 → "박제(영원)" 컨셉 정면 위배.
- 실패 사유 구분이 없어(`failed` 단일) Drive 용량초과(영구) vs 일시 오류(재시도 가능)를 못 가린다.

### 설계 원칙
1. **응답은 즉시**: 논리 레코드 + `pending` 복사본 행만 만들고 바로 반환.
2. **안전망 = Storage 스테이징**: 원본(또는 다운스케일본)을 Supabase Storage 비공개 버킷에 임시 저장 → 팬아웃의 단일 원천. 모든 멤버 synced되면 삭제. (= V-2 영속성 보강을 MVP 안전망으로 일부 선차용)
3. **재시도 가능**: 백그라운드 1차 + cron 2차. `attempts`/`error_code`로 추적, upsert로 멱등.

### 데이터 흐름
```
POST /api/memories/[id]/photos
  1) 멤버십 확인(현행 유지)
  2) (선택) 서버에서 다운스케일 → 원본/축소본을 Storage `photo-staging/{photoId}` 업로드
  3) photos insert (staging_path 세팅, fanout_done=false)
  4) space_members 전원에 photo_drive_copies 'pending' upsert
  5) 즉시 Response.json({ photoId, queued: members.length })
  6) after(() => runFanout(photoId))      ← 응답 후 백그라운드

runFanout(photoId)  (= 개선된 replicatePhoto)
  - staging에서 바이트 로드
  - pending/failed 복사본만 순회, 멤버 토큰으로 Drive 업로드
  - 성공 → synced / 실패 → error_code 분류(quota|auth|other), attempts++
  - 모든 행 synced면 fanout_done=true + staging 삭제

GET /api/cron/photo-fanout  (신규, 하루 수 회 또는 weather cron에 합류)
  - fanout_done=false 인 photos 중 staging_path 있는 것 재시도
  - error_code='quota'/'auth'는 사용자 재연동/정리 전까지 skip(무한 재시도 방지)
  - staging이 N일(예: 7일) 초과 + 여전히 미완 → 정책 결정(보존 연장 vs 포기 알림)
```

### 코드 변경 지점
- `web/lib/photos.ts`
  - `replicatePhoto` → **둘로 분리**: `createPhotoRecord()`(레코드+pending+staging 업로드, 동기) / `runFanout(photoId)`(백그라운드).
  - 복사본 insert를 **upsert(onConflict: photo_id,member_id)** 로 변경(재시도 멱등). `unique(photo_id,member_id)` 이미 존재 → 그대로 활용.
  - Drive 업로드 실패 시 응답코드 파싱: `403 + reason=storageQuotaExceeded → 'quota'`, `401/invalid_grant → 'auth'`, 그 외 `'other'`.
- `web/app/api/memories/[id]/photos/route.ts`
  - `await replicatePhoto` → `createPhotoRecord` 후 `import { after } from "next/server"; after(() => runFanout(photoId))`.
  - Next 16: `after()`는 응답 플러시 후에도 함수 살아있게 함(Vercel에서 동작). 단 **함수 최대 실행시간 내** 끝나야 하므로, 멤버 다수면 cron 2차가 보강.
- `web/lib/google/drive.ts`: `uploadPhotoToDrive` 에서 에러 본문(reason) 추출 헬퍼 추가.
- 신규 `web/app/api/cron/photo-fanout/route.ts` + `vercel.json` cron 추가(또는 기존 weather cron 핸들러에서 함께 처리).

### 스테이징 버킷
- Supabase Storage 비공개 버킷 `photo-staging` 생성(대시보드 1회). 무료 1GB.
- 업로드/삭제는 **service role**(admin)만. 클라이언트 접근 불가.
- 비용: 모든 멤버 synced 후 즉시 삭제하므로 정상 케이스 상주 0. 미연동 멤버가 있으면 그 사진만 잔존(안전망 목적).

### 사용자 피드백(부분 실패 가시화)
- 업로드 응답에 `queued`/즉시 동기 결과 일부를 표시: "사진 올림! (지금 3명에게 복제 중)".
- 모임 상세에서 본인 복사본이 `skipped`(드라이브 미연동)면 *"구글 드라이브를 연결하면 이 사진을 보관할 수 있어요"* 배너(개발기획서 §4.2 게이트 재사용).

### 엣지/주의
- `after()` 실패해도 cron이 줍는다(이중화). 둘 다 best-effort.
- 다운스케일 도입 시 EXIF 회전/원본 화질 보존 정책 결정 필요(원본 보존 우선이면 스테이징은 원본, Drive엔 원본 업로드 — 현행 유지).
- WBS 영향: P4-2/4-4 재작성. P4-2는 ✅였으나 비동기 전환으로 재검증 필요 → 🟡로 회귀 표기 권장.

## A2. 날씨 "서울 폴백" 정직화

### 문제
`ensureWeatherSnapshot`(weather.ts:128)은 좌표를 못 구하면 **조용히 서울 좌표로 조회 후 `source='open-meteo'`** 로 저장한다. 박물관 컨셉에서 *틀린 값을 불변으로 박제*하는 셈("강릉 여행인데 서울 날씨 영구 고정"). 개발기획서 §7은 본래 "좌표 없으면 박제 보류"로 명시.

### 설계 — 3-state로 구분
| 상황 | 처리 | 표시 |
|---|---|---|
| 좌표 있음(place_lat/lng 또는 지오코딩 성공) | 정상 박제 `is_approx=false` | "이날 22° 맑음" |
| place는 있으나 **지오코딩 실패** | **박제 보류**(null 반환, 미저장) → 다음 조회/다음 cron 재시도 | 칩 없음(또는 "날씨 확인 중") |
| place 자체가 없음 | 정책 선택지 ↓ | |

**place 없는 모임** 두 가지 안 — 사용자 결정 필요:
- (안1·권장) 박제 보류. 날씨는 장소가 있어야 의미. 깔끔하고 정직.
- (안2) 서울 기준으로 박제하되 `is_approx=true` + source `open-meteo:approx-seoul`. UI에서 *"(대략) 서울 기준"* 으로 흐리게 표기. 정보는 주되 거짓 단정은 피함.

### 코드 변경 (`web/lib/weather.ts`)
```ts
// 좌표 확정 단계 수정
let lat = memory.place_lat, lng = memory.place_lng, approx = false;
if ((lat == null || lng == null) && memory.place) {
  const g = await geocode(memory.place);
  if (g) { lat = g.lat; lng = g.lng; }
  else return null;                 // 지오코딩 실패 → 보류(재시도)
}
if (lat == null || lng == null) {
  // place 없음: 안1이면 `return null;`
  // 안2이면 ↓
  lat = SEOUL.lat; lng = SEOUL.lng; approx = true;
}
...
await admin.from("weather_snapshots").insert({
  memory_id: memory.id, temp: w.temp, condition: w.condition,
  source: approx ? "open-meteo:approx-seoul" : "open-meteo",
  is_approx: approx,
});
```
- 반환 타입에 `is_approx` 추가 → 모임 상세 칩에서 흐린 톤/괄호 표기.
- `weather_snapshots.is_approx` 컬럼은 §0 마이그레이션에 포함.

### 엣지
- 이미 잘못 저장된 기존 서울 폴백 레코드: 운영 데이터 거의 없으니 무시 가능. 정 필요하면 `source='open-meteo'` AND place 좌표 불일치 행을 1회 정리(보류).
- 불변 원칙(§RLS): `weather_snapshots`는 UPDATE 지양 → 보류분은 **insert 안 함**으로 자연 재시도(현 멱등 구조 유지).

## B4. 1년 전 오늘 회고 (스키마 변경 없음) ⭐컨셉

### 가치
박물관 컨셉의 핵심은 "1년 뒤 다시 꺼내봄"인데 현재는 직접 과거로 스크롤해야 발견. **오늘과 같은 월·일의 과거 박제 셀**을 상단에 노출 → 재방문 트리거.

### 쿼리 (참가 모임만)
```ts
// web/lib/feed.ts 에 추가
export async function listOnThisDay(userId: string) {
  const admin = createSupabaseAdminClient();
  // 내가 참가자인 memory_id
  const { data: parts } = await admin
    .from("memory_participants").select("memory_id").eq("user_id", userId);
  const ids = (parts ?? []).map(p => p.memory_id);
  if (!ids.length) return [];
  // 같은 월-일, 과거 연도
  const { data } = await admin
    .from("memories")
    .select("id, title, date, place")
    .in("id", ids)
    .lt("date", kstToday())
    .filter("date", "not.is", null);
  // 월-일 매칭은 SQL where절(to_char)로 거르는 RPC가 정확.
  return (data ?? []).filter(m => sameMonthDay(m.date, kstToday()));
}
```
- 정확/효율은 **Postgres 함수(RPC)** 권장:
  `where to_char(date,'MM-DD') = to_char((now() at time zone 'Asia/Seoul')::date,'MM-DD') and date < current_date`
  → `memory_participants` 조인으로 본인 것만. 인덱스 부담 적음(모임 수 적은 MVP).

### 표시
- `web/app/(main)/calendar/page.tsx` 상단에 회고 배너: *"📅 N년 전 오늘 — 강릉 여행"* 카드(탭 → `/memory/[id]`).
- 여러 건이면 가로 스크롤. 0건이면 미표시(휑함 방지).
- 연도 차이 계산: `오늘.year - date.year`.

### 엣지
- 2/29: 평년이면 매칭 0(허용) 또는 2/28로 보정(정책). MVP는 매칭 0 허용.
- 타임존 KST 고정(기존 `kstToday` 재사용).
- B5와 연계: 회고가 있으면 알림 1건(`type='on_this_day'`)도 생성 가능(아침 cron).

## B5. 새 활동 알림

### 가치/근거
MVP는 실시간 채팅을 빼고 "비동기 댓글"을 핵심으로 잡았는데(기획서 §5.2), **알림이 없으면 새 댓글/사진을 아무도 모름 → 댓글 기능이 사실상 죽음.** P8-2(친구 5명 실사용) 성패에 직결.

### 단계적 채널 (무료 우선)
| 단계 | 채널 | 인프라 | 권장 |
|---|---|---|---|
| 1 | **인앱 알림함**(notifications 테이블 + 종 아이콘 배지) | 추가 0 | ✅ 지금 |
| 1b | **Supabase Realtime** 구독으로 배지 실시간 갱신 | 무료 포함 | ✅ 같이 |
| 2 | **이메일 다이제스트**(하루 1회 요약) | Resend 등 무료티어 + 도메인 검증(승인 필요) | 선택 |
| 3 | **Web Push**(PWA, 가장 앱다움) | 서비스워커+VAPID+권한 | 후순위 |

→ **1+1b를 MVP로**, 2/3은 이후. 외부 의존 없이 컨셉에 맞고 즉시 가치.

### 1단계 구현 (인앱)
**생성(팬아웃):** 댓글/사진 추가 시, 행위자 제외 참가자 전원에 알림 row.
- `web/lib/comments.ts`(또는 `app/actions/comments.ts`) 댓글 insert 직후:
```ts
async function notifyParticipants(memoryId, actorId, type, preview) {
  const admin = createSupabaseAdminClient();
  const { data: parts } = await admin
    .from("memory_participants").select("user_id").eq("memory_id", memoryId);
  const rows = (parts ?? [])
    .filter(p => p.user_id !== actorId)
    .map(p => ({ user_id: p.user_id, memory_id: memoryId, actor_id: actorId, type, preview }));
  if (rows.length) await admin.from("notifications").insert(rows);
}
```
- 사진: `runFanout`/업로드 완료 지점에서 `type='photo'` 동일 호출(중복 방지 위해 사진 1건당 1회).

**조회/표시:**
- `web/lib/notifications.ts`: `listMyNotifications()`, `unreadCount()`, `markRead(ids)`.
- 하단 탭(또는 헤더)에 종 아이콘 + 미읽음 배지. 탭 → 알림 목록(각 항목 탭 시 `/memory/[id]` 이동 + read 처리).
- RLS: 본인 select/update만(§0). insert는 admin 전용.

**1b 실시간:**
- 클라이언트에서 `supabase.channel('notif:'+userId).on('postgres_changes', { event:'INSERT', table:'notifications', filter:`user_id=eq.${userId}` }, ...)`로 배지 즉시 증가. Supabase Free Realtime 포함.

### 2단계 이메일 다이제스트(선택, 승인 후)
- `GET /api/cron/digest`(매일 아침 KST): user별 미읽음 알림 group → "어제 강릉 여행에 사진 3장, 댓글 2개"를 1통으로. Resend 무료(월 3k통)·발신 도메인 검증 필요(=사용자 작업/승인 항목).
- 환경변수 `RESEND_API_KEY` 추가. 도메인 없으면 보류.

### 엣지
- 알림 폭주 방지: 같은 모임 같은 type 다건은 표시에서 묶기("사진 3장"). 저장은 건별이어도 UI 집계.
- 행위자 본인 제외 필수.
- 탈퇴/제외된 참가자: `on delete cascade`로 정리됨.

## 결정 사항 (2026-06-11 확정 — 추천안 채택)

1. **A1 스테이징 = 원본 보존.** 컨셉이 "박제(영원·고정)"이고 Drive 복사본도 원본이므로 안전망/지연합류분도 원본 화질이어야 일관. 스테이징은 **전원 synced 시 삭제**되어 정상 케이스 상주 0 → 무료 1GB로 충분(MVP 규모 작음). 다운스케일 영구백업(V-2)은 별도 후속.
2. **A2 = 박제 보류(안1).** place 없는 모임엔 날씨를 박제하지 않는다(날씨는 장소가 있어야 의미). 거짓 단정 회피 + 구조 단순. `is_approx` 컬럼은 만들어 두되(지오코딩이 광역만 잡힌 경우 대비) place-less는 보류로 처리.
3. **B5 = 인앱(1+1b)만 우선.** notifications 테이블 + Supabase Realtime 배지까지. 이메일 다이제스트(2단계)는 도메인/Resend 승인 필요 → **보류**(추후 별도 결정).

> 위 확정에 따른 코드 영향: A2는 §A2 코드의 "place 없음" 분기를 `return null`(보류)로, `is_approx`는 지오코딩 광역 매칭 등에만 사용. A1·B5는 본문 설계 그대로. 마이그레이션 `0003`은 §0 그대로 유효(컬럼/테이블 변동 없음).
