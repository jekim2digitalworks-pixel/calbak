## 0. 유의사항 (작업 전 반드시 숙지)

사용자가 명시적으로 기록을 요청한 항목 + 원 기획서 §12의 주의점을 통합한다.

### 0.1 결정 상태 표기 규칙
- `[확정]` 합의 사항. 변경 시 설계 전반에 영향 → 바꾸려면 의사결정 로그부터 갱신.
- `[제안]` 권고안. 사용자 판단으로 조정 가능.
- `[미정]` 결론 필요. **구현 전 반드시 결론**을 낼 것.
- **`[확정]`·`[미정]`·`[제안]` 무관하게, 진행 중 애매한 부분이 생기면 임의로 정하지 말고 사용자에게 의견을 구한다.** (← 사용자 핵심 지시)

### 0.2 작업 방식
1. **기능을 한 번에 다 만들지 않는다.** 아래 Phase/Task 단위로 쪼개 진행한다(한 번에 9개 기능 요청 금지 — 품질·일관성 저하).
2. **권장 순서**: 데이터 모델 → 백엔드 API(모임 CRUD) → 캘린더 화면 → 모임 상세(사진·댓글) → 날씨 박제 → 친구 초대 → 캘린더 연동.
3. **`[미정]` 항목은 구현 전 결론.** (현재 모든 MVP `[미정]`은 의사결정 로그에서 해소됨 — 새 `[미정]`이 생기면 이 원칙 적용.)

### 0.3 디자인(supanova) 오용 주의
- supanova는 **랜딩페이지 특화** 스킬. 랜딩에는 풀 적용하되, **앱 본체에는 다크/글래스모피즘 비주얼을 쓰지 않는다.**
- 앱 본체는 supanova의 **태도(원칙)만** 차용: Pretendard, `word-break: keep-all`, 모바일 우선, 디테일 집착, 출력 생략 금지(full-output).
- 앱 컬러: **웜화이트/크림 베이스 + 부드러운 그림자 + 저채도 액센트 1~2색** (따뜻하고 아늑한 추억 앱). 레퍼런스: Day One, Notion Calendar.
- 지시할 때마다 "따뜻한 톤" 제약을 **반복 명시**해 다크 비주얼 혼입 방지.

### 0.4 컨셉 일관성 점검 질문
- 새 기능을 추가할 때마다 자문: **"이게 강(흘러감)인가, 박물관(고정)인가?"** 강이면 캘박이 아니다.
- 박제된 값(날씨 등)은 가급적 **불변 스냅샷**으로 다룬다.

### 0.5 이 프로젝트 특수 제약
- **무료 티어 우선**: 출시 전까지 Vercel + Supabase 무료 스택 유지. 유료 전환은 사용자 승인 후.
- **사진은 멤버별 Google Drive에 복제 저장**(우리 저장 비용 0). 사진 기능은 Drive 연동 필수(A안).
- **민감 정보(구글 refresh token)** 는 클라이언트에 노출 금지 — 서버 Route Handler + service role 전용.
