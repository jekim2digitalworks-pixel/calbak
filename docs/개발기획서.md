## 3. 아키텍처 개요

```
[모바일 웹 브라우저]
        │
        ▼
[Next.js on Vercel] ── 서버 컴포넌트 / Route Handlers (app/api/*)
        │                     │
        │ (RLS, anon key)     │ (service role, server-only)
        ▼                     ▼
[Supabase]            [Google APIs]
 ├ Auth (Google)       ├ Drive v3   ← 사진 복제 팬아웃
 ├ Postgres + RLS      └ Calendar v3 ← 모임 일정 푸시
 └ (Storage: v1.1 백업용)
        ▲
        │
[Open-Meteo Archive API]  ← 날씨 박제(서버에서 호출)
```

- 클라이언트는 Supabase **anon key**로 RLS 보호된 데이터만 접근.
- Drive 복제 팬아웃, Calendar 푸시, 토큰 사용, 날씨 박제 등 **민감/특권 작업은 Route Handler에서 service role + 서버 보관 토큰으로** 수행.
- 리프레시 토큰은 클라이언트에 절대 노출하지 않는다.

## 4. 인증 & Google 연동 설계

### 4.1 가입/로그인 플로우
1. 사용자가 "Google로 시작" → Supabase Auth Google OAuth.
2. 요청 스코프(최소 권한 원칙):
   - `openid email profile` (기본)
   - `https://www.googleapis.com/auth/drive.file` — **앱이 만든 파일만** 접근(사용자의 기존 Drive 파일은 못 봄)
   - `https://www.googleapis.com/auth/calendar.events` — **앱이 만든 일정만** 관리
   - `access_type=offline`, `prompt=consent` 로 refresh token 확보
3. 콜백에서 granted scope를 검사해 `drive_connected` / `calendar_connected`를 기록(**연동 여부 체크**). 사용자가 일부 권한을 거부해도 로그인은 성립.
4. refresh token은 암호화하여 `user_google_connections`에 저장(서버 전용).

### 4.2 연동 게이트
- **사진 업로드/조회**: `drive_connected = true` 필요(D3-A). 미연동 시 "구글 드라이브를 연결하면 사진을 보관/열람할 수 있어요" 안내 + 재연동 버튼.
- **캘린더 푸시**: `calendar_connected = true`일 때만 모임을 GCal에 반영. 미연동이어도 캘박 내 일정 기능은 정상.

## 5. 사진 저장 설계 — 멤버별 Google Drive 복제

### 5.1 핵심 원리
사진을 한 곳(예: owner)이 아니라 **공유받기를 신청(=Drive 연동)한 모든 멤버의 Drive에 복사본으로 분산 저장**한다. 각 멤버는 *자기 Drive의 자기 복사본*을 본다.

→ 누가 지워도 내 Drive에 남아 있으면 나는 계속 본다. **단일 의존점이 사라져 "박제(영원히 고정)" 컨셉이 성립**한다.

### 5.2 업로드 = 팬아웃 복제
1. 멤버 B가 사진 업로드 → Route Handler가 파일 수신.
2. 해당 공간에서 `drive_connected = true`인 모든 멤버에 대해, 각자의 refresh token으로 각자 Drive의 `캘박/{공간명}` 폴더에 복사본 업로드.
3. `photos`에 논리 레코드 1건, `photo_drive_copies`에 멤버별 물리 복사본(`drive_file_id`, `thumbnail_link`, `sync_status`) 기록.
4. 썸네일은 Drive가 자동 생성(`thumbnailLink`)하여 별도 생성 불필요.

### 5.3 표시 / 삭제 / 백필
- **표시**: 각 멤버는 자기 `drive_file_id` 복사본을 자기 토큰으로 로드(서버 프록시 `/api/photos/[id]`). 공개 공유 불필요 → 프라이버시 유지.
- **삭제**: 한 멤버가 지워도 다른 멤버 복사본은 보존. 복사본이 하나라도 남으면 사진은 살아있음.
- **나중 합류**: 신규 멤버가 Drive 연동 시 기존 사진을 본인 Drive로 일괄 **백필 복제**.
- **권한 스코프**: `drive.file` → 앱이 만든 파일만 접근. 멤버의 다른 Drive 파일은 안전.

### 5.4 리스크 / 한계 (기록용)
- **용량**: 멤버 개인 15GB(Gmail·Google Photos 공유) 소비. 가득 차면 그 멤버 복제 실패 → graceful skip + 알림.
- **복제 지연**: 업로드당 N개 복사 → 비동기 처리 권장(즉시 1차 응답 후 백그라운드 팬아웃, `sync_status`로 추적).
- **완전 영속성**은 아님: 모든 멤버가 동시에 지우면 소실. v1.1에서 저해상 썸네일 Supabase 백업(D10)으로 보강.

## 6. 구글 캘린더 연동 설계 `[추가]`

원 기획서에 없던 추가 요구. **방향: 캘박 → Google Calendar (단방향 푸시)** 를 MVP 기본값으로 한다.

- 모임(특히 미래 일정) 생성/수정 시, `calendar_connected`인 참여 멤버의 Google Calendar에 이벤트 생성/갱신.
- 멤버별 `google_event_id`를 `memory_calendar_events`에 저장해 수정·삭제 동기화.
- 이벤트 본문에 캘박 모임 딥링크 포함("캘박에서 열기").
- **읽기 방향**(개인 GCal 일정을 캘박 그리드에 표시)은 범위·프라이버시 고려해 **v1.1 [제안]** 으로 보류.

## 7. 날씨 박제 설계 (Open-Meteo Archive)

- 출처: `https://archive-api.open-meteo.com/v1/archive` (키 불필요, 무료, 전 세계, 시간 단위, 1940~).
- **예보가 아니라 그날의 실제 날씨를 과거 기록으로 박제**(기획서 6-(5)).
- 미래 날짜 모임: 날짜가 **지난 뒤** 최초 접근 시 또는 Vercel Cron(일 1회)으로 조회하여 박제.
- 한 번 박제되면 **불변 스냅샷**(`weather_snapshots`)으로 다룬다(기획서 §8 스냅샷 패턴).
- 좌표: 모임 `place`의 위경도(지오코딩) 또는 공간 기본 좌표. 좌표 없으면 박제 보류.
- 착수 전 실제 응답 1건 검증 필요(WBS P5 선행 작업).

## 8. 데이터 모델 (Postgres + RLS)

초기 스키마는 [`/supabase/migrations/0001_init.sql`](../supabase/migrations/0001_init.sql) 참조. 테이블 개요:

| 테이블 | 역할 | 핵심 컬럼 |
|---|---|---|
| `profiles` | 사용자 프로필(↔ auth.users) | id, nickname, avatar_url |
| `user_google_connections` | 구글 토큰·연동 상태(서버 전용) | user_id, refresh_token_enc, drive_connected, calendar_connected, drive_folder_id |
| `spaces` | 캘박 공간 | id, name, created_by |
| `space_members` | 멤버십(RLS 기준점) | space_id, user_id, role |
| `memories` | 모임 = 박제 셀 | id, space_id, title, **date**, place, created_by |
| `weather_snapshots` | 박제된 날씨(불변) | memory_id, temp, condition, source, fetched_at |
| `photos` | 논리적 사진 | id, memory_id, uploaded_by, taken_at, caption |
| `photo_drive_copies` | 멤버별 물리 복사본 | photo_id, member_id, drive_file_id, thumbnail_link, sync_status |
| `comments` | 비동기 댓글 | id, memory_id, author_id, content |
| `memory_calendar_events` | 모임↔GCal 매핑 | memory_id, member_id, google_event_id, sync_status |
| `invites` | 초대 링크/코드 | id, space_id, code, expires_at, max_uses |
| `diary_entries` *(v1.1)* | 개인 비밀 일기 | user_id, date, content |

### RLS 원칙
- 공간 종속 테이블(`memories`/`photos`/`comments`/…): **자신이 멤버인 공간**의 행만 read/write (`EXISTS (space_members where user_id = auth.uid())`).
- 멤버 개인 테이블(`photo_drive_copies`/`memory_calendar_events`/`user_google_connections`): `member_id/user_id = auth.uid()` 인 행만.
- 토큰 등 민감 컬럼은 **클라이언트가 직접 쿼리하지 않으며**, 서버 Route Handler가 service role로만 접근.
- 박제 컨셉상 `weather_snapshots`는 사실상 불변(UPDATE 지양).
