## 1. 한 줄 정의

`[확정]` 캘박은 **친한 친구들끼리 추억과 일정을 날짜 단위로 박제하는 프라이빗 공간**이다.

핵심 비유로 정체성을 고정한다.

- 카카오톡은 **강(江)** 이다. 모든 대화가 시간순으로 흘러가 사라지고, 작년 여행 이야기는 스크롤 수천 번 위로 묻힌다.
- 캘박은 **박물관** 이다. 모든 추억이 그 날짜 칸에 못 박혀, 1년 뒤에 열어도 그 자리에 그대로 있다.

이 "흘러가지 않고 고정된다"는 성질이 캘박의 전부다. 모든 기능 결정은 이 기준에 부합하는지로 판단한다.

---

## 2. 포지셔닝

`[확정]` **A안 채택: 카카오톡을 대체하지 않는다. 카톡 옆에 추가로 두는 "추억 전담 별채"다.**

(1) 캘박은 메인 메신저가 되려 하지 않는다. "오늘 저녁 뭐 먹지?" 같은 흘러가는 잡담은 카톡의 영역으로 남긴다.

(2) 캘박은 **박제할 가치가 있는 순간**(여행, 모임, 기념일)만 가져온다. 카톡이 구조적으로 못 하는 일(추억 보관)을 독점한다.

(3) 따라서 친구들을 카톡에서 "이사"시킬 필요가 없다. 카톡과 함께 쓰는 보조 앱으로 포지셔닝한다.

근거: 친한 친구 5명을 동시에 메인 메신저에서 빼내는 전환 비용은 개인 프로젝트가 이길 수 없다. 또한 모든 대화를 떠안으려 하면 "더 못난 카톡"이 되어 사용자가 카톡으로 회귀한다. 카톡과 싸우지 않고 카톡이 버리는 것을 줍는 전략이 완주율과 차별성 모두에서 유리하다.

---

## 3. 핵심 차별점

`[확정]` **대화와 기록이 "사람"이 아니라 "날짜"에 매달려 있다.**

(1) 카톡 대화방은 사람·그룹에 종속된다. 대화는 끝없이 흐르고 과거는 검색으로만 겨우 닿는다.

(2) 캘박은 사진, 영수증, 댓글, (v2의) 채팅이 모두 **특정 날짜의 모임 칸**에 종속된다. "작년 10월 제주도" 칸을 열면 그날의 모든 것이 통째로 박제되어 있다.

이 한 가지가 카톡이 구조상 절대 흉내 낼 수 없는 지점이며, 캘박의 존재 이유다.

## 4. 타깃과 사용 시나리오

(1) `[제안]` 1차 타깃: 자주 못 보지만 추억은 쌓고 싶은 **소규모 친밀 그룹**(대학 동기, 오랜 친구, 가족 등 3~8명).

(2) 대표 시나리오
- 모임 전: 친구 중 한 명이 "11월 22일 강릉 여행" 일정을 미리 박제(생성)하고 친구들을 초대한다.
- 모임 후: 그날 칸에 각자 찍은 사진을 올리고, 영수증을 등록하고, "그때 그 횟집 진짜였다" 같은 댓글을 남긴다.
- 시간이 흐른 뒤: 캘린더를 열어 과거 칸을 보며 그날의 사진·날씨·대화를 다시 본다. 1년 뒤에도 그 칸에 새 댓글을 달 수 있다.

## 5. MVP 범위

MVP의 목표는 "친구 5명이 일주일 동안 실제로 써보는 실험"이 가능한 최소 제품이다.

### 5.1 MVP에 포함 (In Scope)

1. 전체화면 캘린더 + 날짜 상호작용
2. 모임(박제 셀) 생성·조회
3. 모임에 사진 업로드
4. 모임에 댓글 작성 (비동기, 실시간 아님)
5. 과거 날씨 박제
6. 개인 일기 레이어
7. 친구 초대 및 공간 공유

### 5.2 MVP에서 제외 — v2로 연기 (Out of Scope)

1. `[확정]` **실시간 채팅**: 웹소켓 서버, 푸시 인프라, 메시지 동기화는 MVP 전체를 합친 것보다 무겁다. MVP에서는 "비동기 댓글"로 대체하고, 실제 사용이 확인되면 v2에서 실시간으로 승격한다. 박물관 컨셉상 비동기 댓글이 오히려 더 어울린다.

2. `[제안]` **N빵 정산 자동화 + 카톡 자동 전송**: 정산은 돈이 엮여 정확성 부담이 크고, 카카오 메시지 API는 정책·심사 허들이 있다. MVP에서는 "정산 결과를 이미지/텍스트로 만들어 사용자가 직접 카톡에 붙여넣기" 수준까지만. 자동 전송은 v2.

3. 다중 공간(여러 친구 그룹) 동시 운영: MVP는 1인 1공간으로 단순화 가능. (8. 데이터 모델 참조)

## 6. 기능 명세

### (1) 전체화면 캘린더 `[확정]`
- 월 단위 전체화면 그리드. 모바일 우선.
- 모임이 박제된 날짜에는 시각적 표시(점, 썸네일 등).
- 날짜 탭(모바일) / 호버(데스크톱) 시 해당 날의 요약(모임명, 사진 썸네일, 날씨)을 미리 보여준다.
- 날짜 선택 시 해당 날의 상세(박제 셀)로 진입.

### (2) 모임 = 박제 셀 `[확정]`
- 하나의 날짜에 하나 이상의 모임을 박제한다.
- 모임 속성: 제목, 날짜, 장소(선택), 참여자, 사진들, 댓글들, 날씨, (선택) 영수증.
- 미래 날짜에도 미리 생성 가능(일정). 그날이 지나면 자연스럽게 추억으로 전환된다. 미래/과거를 별도 모드로 가르지 않고 같은 셀이 시간에 따라 의미만 바뀐다.

### (3) 사진 `[확정]`
- 모임 단위로 여러 명이 사진을 올린다.
- `[제안]` 저장소는 AWS S3. 원본 + 썸네일 분리 저장. 업로드는 백엔드 경유 또는 presigned URL 방식.

### (4) 댓글 (비동기) `[확정]`
- 모임 칸에 시간순으로 쌓이는 댓글. 카톡 채팅이 아니라 인스타 게시물 댓글에 가깝다.
- 1년 뒤에도 같은 칸에 새 댓글을 달 수 있다. 이것이 "날짜에 매달린 대화"의 MVP 구현체다.

### (5) 날씨 박제 `[확정 / 일부 미정]`
- `[확정]` 예보(미래)가 아니라 **그날의 실제 날씨를 기록(과거)으로 박제**한다. 날씨가 기억의 트리거가 되게 한다. (예: "이날 18도 맑음, 벚꽃 절정")
- `[미정]` 과거 날씨 데이터 출처. 후보: 기상청 공공 API, Open-Meteo Historical API. 무료 과거 날씨 API는 지역·기간 제한이 있을 수 있어 MVP 착수 전 실제 응답을 확인해야 한다. (이 부분은 확인 전까지 추측이며 검증 필요)

### (6) 개인 일기 레이어 `[제안]`
- 한 날짜 칸에 두 개의 레이어를 둔다.
  - 공용 기록: 친구 모두가 보는 사진·댓글.
  - 비밀 일기: 나만 보는 개인 기록.
- 같은 날을 친구들과 다르게 기억할 수 있게 한다. 단순 공유 앨범과 차별화되는 지점이다.
- MVP 비용이 부담되면 v1.1로 미뤄도 무방.

### (7) 친구 초대 및 공유 `[확정]`
- 한 사용자가 공간(캘박)을 만들고 친구를 초대한다.
- 초대된 친구는 같은 캘린더·모임·사진·댓글을 공유한다.
- `[제안]` 초대는 링크 기반(초대 코드/URL). 별도 친구 검색 시스템은 MVP에 불필요.

### (8) [v2] 실시간 채팅
- 날짜별 채팅방 + 알림으로 해당 채팅방 이동. MVP의 비동기 댓글이 검증된 뒤 승격.

### (9) [v2] N빵 정산
- 영수증 기반 자동 분할 계산 → 결과 카드 생성. 카톡 자동 전송은 정책 검토 후.

## 7. 화면 구성과 흐름

`[제안]` MVP 핵심 화면은 4개로 충분하다.

1. **메인: 전체화면 캘린더** — 앱을 열면 가장 먼저 보이는 화면. 박제된 날짜가 한눈에 보인다.
2. **모임 상세(박제 셀)** — 날짜를 선택하면 진입. 사진, 날씨, 댓글, (있으면) 영수증, 비밀 일기 진입점.
3. **모임 생성/편집** — 제목·날짜·장소·참여자 입력.
4. **공간 관리** — 친구 초대(링크 생성), 멤버 목록.

`[미정]` 메인 진입 화면을 "전체화면 캘린더"로 할지 "최근 박제된 모임 피드"로 할지. 본 문서는 컨셉("박물관")에 더 충실한 **전체화면 캘린더**를 기본값으로 둔다. 다만 빈 캘린더는 초기 사용자에게 휑할 수 있으므로, 모임이 적을 때는 피드형 안내를 함께 노출하는 절충안을 권한다.

## 8. 데이터 모델 (엔티티 초안)

`[제안]` 주은님 스택(JPA/QueryDSL/MySQL)에 맞춘 초안. 관계 위주로 기술한다.

1. **User** — 사용자. (id, nickname, ...)

2. **Space** — 하나의 캘박(친구 그룹의 프라이빗 공간). 여러 User가 속한다.
   - User ↔ Space: 다대다 (멤버십). 중간 엔티티 `SpaceMember`(role 포함) 권장.
   - MVP에서 다중 공간이 부담되면 "User 1명당 Space 1개"로 단순화 가능.

3. **Memory (모임/박제 셀)** — 특정 날짜에 박제되는 핵심 엔티티.
   - Space 1 : N Memory
   - 속성: title, date, place(선택), createdBy, weatherSnapshot(선택)
   - 핵심: `date`가 이 엔티티의 정체성. 캘린더 그리드와 직접 연결된다.

4. **Photo** — 모임에 속한 사진.
   - Memory 1 : N Photo
   - 속성: s3Key, thumbnailKey, uploadedBy, takenAt(선택)

5. **Comment** — 모임에 달리는 비동기 댓글.
   - Memory 1 : N Comment
   - 속성: author, content, createdAt

6. **DiaryEntry** — 개인 비밀 일기.
   - User + date 단위. Memory와 분리(모임 없는 날에도 일기 가능).
   - 가시성: 작성자 본인만.

7. **WeatherSnapshot** — 박제된 날씨.
   - Memory에 1:1 또는 임베디드. 속성: temp, condition, source, fetchedAt
   - 한 번 박제되면 변하지 않는 값(스냅샷)으로 다룬다.

8. **[v2] Receipt / Settlement** — 영수증·정산. MVP 제외.

설계 노트: "흘러가지 않고 고정"이라는 컨셉상, WeatherSnapshot처럼 한 번 기록되면 외부 값이 바뀌어도 그대로 보존되는 **스냅샷 패턴**이 캘박 전반에 어울린다. 평소 상태를 UPDATE 하는 방식과 달리, 박제된 값은 가급적 불변으로 다루는 것을 권한다.

## 9. 기술 스택 제안

`[제안]` 주은님의 강점과 MVP 속도를 함께 고려한 안. 모두 조정 가능.

1. **백엔드**: Spring Boot + JPA + QueryDSL + MySQL. (주은님 주력 스택. 캘린더 범위 조회, 날짜별 집계에 QueryDSL 동적 쿼리가 잘 맞는다.)

2. **인프라**: AWS. 사진 저장은 S3.

3. **프론트엔드**: 모바일 우선 반응형 웹. "앱에 특화"라는 요구를 MVP 단계에서 네이티브로 가면 비용이 급증하므로, 먼저 **모바일 웹(PWA 후보)** 으로 만들고 홈 화면 추가·푸시는 이후 단계에서 검토한다.

4. **스타일**: Tailwind CSS + Pretendard 폰트.

트레이드오프: 네이티브 앱(React Native/Flutter)은 카메라·푸시·홈 위젯에서 강점이 있으나 MVP 완주를 늦춘다. "친구 5명이 일주일 써보는 실험"이 1차 목표이므로 모바일 웹으로 빠르게 검증 후 결정하는 것을 권한다. (이 판단은 MVP 우선이라는 전제에 근거하며, 네이티브가 목적이라면 달라진다.)

## 10. UI/UX 가이드 — supanova 사용법 (중요)

supanova-design-skill 레포를 직접 확인한 결과에 근거한 가이드다. 결론부터: **supanova는 "절반만" 맞는다. 쓸 곳과 쓰지 말 곳을 구분해야 한다.**

### 10.1 supanova의 정체

레포 README가 명시하듯 supanova는 **랜딩페이지 특화** 스킬이다("일반 웹앱이 아닌 전환율 중심 랜딩페이지 패턴"). 즉 전환율 중심의 소개 페이지를 $150k 에이전시 수준으로 만드는 데는 탁월하지만, 손가락으로 만지는 **인터랙티브 앱 UI**(캘린더 호버, 날짜 탭, 사진 업로드)를 만드는 스킬은 아니다.

### 10.2 적용 원칙

1. **랜딩/소개 페이지** → supanova를 그대로 사용. `taste-skill` + `output-skill` 조합 권장. 제값을 한다.

2. **앱 본체 UI** → supanova의 **시각 자산(다크·글래스모피즘·하이엔드)을 그대로 쓰지 말 것.** 캘박의 정서는 추억·따뜻함인데, 다크 글래스 톤은 자칫 "차가운 추억 앱"이 된다. 대신 supanova가 강조하는 **보편 원칙(태도)** 만 가져온다.
   - Pretendard 폰트
   - `word-break: keep-all` (한국어 문장 가독성)
   - 모바일 우선
   - 디테일에 대한 집착, 출력 생략 금지(full-output)

3. **앱 본체 컬러 방향** `[제안]`: 다크 하이엔드 대신 **웜화이트/크림 베이스 + 부드러운 그림자 + 저채도 액센트 1~2색.** 레퍼런스로 Day One(따뜻한 다이어리 앱), Notion 캘린더(절제된 인터랙션)를 권한다.

### 10.3 Claude CLI에 줄 지시 예시

랜딩페이지 작업 시:
> supanova-design-skill의 taste-skill과 output-skill을 참조해서, 캘박을 소개하는 랜딩페이지를 만들어줘. 단일 HTML, Tailwind CDN, Pretendard.

앱 본체 작업 시:
> supanova의 다크/글래스모피즘 비주얼은 쓰지 마. 대신 supanova의 원칙(Pretendard, word-break:keep-all, 모바일 우선, 디테일, 출력 생략 금지)만 따라줘. 컬러는 웜화이트/크림 베이스에 저채도 액센트로, 따뜻하고 아늑한 추억 앱 느낌으로 만들어줘.

## 12. Claude CLI 작업 가이드 / 주의점

1. **기능을 한 번에 다 시키지 말 것.** "6. 기능 명세"의 항목 단위로 쪼개 지시한다. 한 번에 9개 기능을 요청하면 출력이 길어지고 품질·일관성이 떨어진다.

2. **권장 작업 순서**: 데이터 모델/엔티티 → 백엔드 API(모임 CRUD) → 캘린더 화면 → 모임 상세(사진·댓글) → 날씨 박제 → 친구 초대.

3. **supanova 오용 주의** (10절): 앱 본체에 다크 글래스 비주얼이 섞여 들어오지 않도록, 지시할 때마다 "따뜻한 톤" 제약을 반복해 명시한다.

4. **`[미정]` 항목은 구현 전에 결론을 낼 것**: 과거 날씨 API 출처(5-(5)), 메인 진입 화면(7절), 다중 공간 여부(8절).

5. **컨셉 일관성 점검 질문**: 새 기능을 추가할 때마다 "이게 강(흘러감)인가 박물관(고정)인가?"를 자문한다. 강이면 캘박이 아니다.
